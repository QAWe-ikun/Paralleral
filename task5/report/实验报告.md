## 2. 实验过程和核心代码

### 2.1 通用矩阵乘法（GEMM）

#### 2.1.1 算法原理

通用矩阵乘法的公式为：

$$C = A \times B$$

$$C_{m,k} = \sum_{n=1}^{N} A_{m,n} \times B_{n,k}$$

其中 A 是 M×N 矩阵，B 是 N×K 矩阵，C 是 M×K 矩阵。

串行算法的时间复杂度为 $O(M \times N \times K)$，对于大规模矩阵（如 2048×2048），计算量达到约 $10^{10}$ 次浮点运算，非常适合并行化。

#### 2.1.2 并行策略

**按行分解**：将矩阵 C 的每一行分配给不同线程计算。

- 线程 t 计算 C 的第 t 行：$C_{t,:} = A_{t,:} \times B$
- 各行之间无数据依赖，无需同步
- 每个线程写入 C 的不同行，无竞争条件

```c
// 串行版本
for (int i = 0; i < M; i++) {
    for (int j = 0; j < K; j++) {
        float sum = 0.0f;
        for (int p = 0; p < N; p++) {
            sum += A[i * N + p] * B[p * K + j];
        }
        C[i * K + j] = sum;
    }
}
```

#### 2.1.3 B 矩阵转置优化

**问题**：原始实现中 `B[p * K + j]` 在内层循环中是列优先访问，导致缓存命中率低。

**解决方案**：预先将 B 矩阵转置为 $Bt(Bt[j][p] = B[p][j])$，将列访问转换为行访问。

```c
// 转置函数
void transpose_B(float *B, float *Bt, int N, int K) {
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < K; j++) {
            Bt[j * N + i] = B[i * K + j];
        }
    }
}

// 优化后的 functor
void *gemm_functor(int idx, void *args) {
    gemm_args_t *gemm_args = (gemm_args_t *)args;
    float *A = gemm_args->A;
    float *Bt = gemm_args->Bt;  // 使用转置后的 B
    float *C = gemm_args->C;
    int N = gemm_args->N;
    int K = gemm_args->K;

    for (int j = 0; j < K; j++) {
        float sum = 0.0f;
        for (int p = 0; p < N; p++) {
            sum += A[idx * N + p] * Bt[j * N + p];  // 连续内存访问
        }
        C[idx * K + j] = sum;
    }
    return NULL;
}
```

**优化效果**：内层循环中 $Bt[j][p]$ 是连续内存访问，缓存命中率提升。

### 2.2 OpenMP 矩阵乘法

#### 2.2.1 实现代码

```c
// 默认调度
void gemm_omp_default(float *A, float *B, float *C, int M, int N, int K,
                      int num_threads) {
#pragma omp parallel for num_threads(num_threads) collapse(2)
    for (int i = 0; i < M; i++) {
        for (int j = 0; j < K; j++) {
            float sum = 0.0f;
            for (int p = 0; p < N; p++) {
                sum += A[i * N + p] * B[p * K + j];
            }
            C[i * K + j] = sum;
        }
    }
}

// 静态调度
void gemm_omp_static(float *A, float *B, float *C, int M, int N, int K,
                     int num_threads) {
#pragma omp parallel for num_threads(num_threads) schedule(static, 1) collapse(2)
    for (int i = 0; i < M; i++) {
        for (int j = 0; j < K; j++) {
            float sum = 0.0f;
            for (int p = 0; p < N; p++) {
                sum += A[i * N + p] * B[p * K + j];
            }
            C[i * K + j] = sum;
        }
    }
}

// 动态调度
void gemm_omp_dynamic(float *A, float *B, float *C, int M, int N, int K,
                      int num_threads) {
#pragma omp parallel for num_threads(num_threads) schedule(dynamic, 1) collapse(2)
    for (int i = 0; i < M; i++) {
        for (int j = 0; j < K; j++) {
            float sum = 0.0f;
            for (int p = 0; p < N; p++) {
                sum += A[i * N + p] * B[p * K + j];
            }
            C[i * K + j] = sum;
        }
    }
}
```

#### 2.2.2 调度策略说明

| 调度策略 | 指令 | 说明 |
|---------|------|------|
| 默认 | 无 | 编译器决定调度方式 |
| 静态 | `schedule(static, 1)` | 预先平均分配，每行一个块 |
| 动态 | `schedule(dynamic, 1)` | 运行时动态分配，每行一个块 |

### 2.3 Pthreads parallel_for 动态链接库

#### 2.3.1 API 设计

```c
// 调度策略枚举
typedef enum {
    SCHEDULE_STATIC,   // 静态调度
    SCHEDULE_DYNAMIC,  // 动态调度
    SCHEDULE_GUIDED    // 引导调度
} schedule_type_t;

// 并行配置结构体
typedef struct {
    int num_threads;       // 线程数
    schedule_type_t schedule; // 调度策略
    int chunk_size;        // 块大小
} parallel_config_t;

// 基础 API（静态调度）
int parallel_for(int start, int end, int increment,
                 void *(*functor)(int, void *),
                 void *arg, int num_threads);

// 高级 API（支持调度策略）
int parallel_for_advanced(int start, int end, int increment,
                          void *(*functor)(int, void *),
                          void *arg, parallel_config_t *config);
```

#### 2.3.2 静态调度实现

```c
static void *thread_work_static(void *arg) {
    thread_work_t *work = (thread_work_t *)arg;
    for (int i = work->start; i < work->end; i += work->increment) {
        work->functor(i, work->arg);
    }
    return NULL;
}

// 静态调度：预先平均分配任务
int base_chunk = total_iterations / num_threads;
int remainder = total_iterations % num_threads;

int current_start = start;
for (int t = 0; t < num_threads; t++) {
    int chunk = base_chunk + (t < remainder ? 1 : 0);
    int current_end = current_start + chunk * increment;
    
    works[t].start = current_start;
    works[t].end = current_end;
    // ...
    current_start = current_end;
}
```

#### 2.3.3 动态调度实现

```c
static pthread_mutex_t dynamic_mutex = PTHREAD_MUTEX_INITIALIZER;
static int dynamic_next_start = 0;

static void *thread_work_dynamic(void *arg) {
    thread_work_t *work = (thread_work_t *)arg;
    int local_start, local_end;

    while (1) {
        pthread_mutex_lock(&dynamic_mutex);
        if (dynamic_next_start >= dynamic_end) {
            pthread_mutex_unlock(&dynamic_mutex);
            break;
        }
        local_start = dynamic_next_start;
        local_end = local_start + dynamic_chunk_size * dynamic_increment;
        if (local_end > dynamic_end) {
            local_end = dynamic_end;
        }
        dynamic_next_start = local_end;
        pthread_mutex_unlock(&dynamic_mutex);

        for (int i = local_start; i < local_end; i += dynamic_increment) {
            work->functor(i, work->arg);
        }
    }
    return NULL;
}
```

#### 2.3.4 引导调度实现

```c
static void *thread_work_guided(void *arg) {
    thread_work_t *work = (thread_work_t *)arg;
    int local_start, local_end, chunk;

    while (1) {
        pthread_mutex_lock(&guided_mutex);
        if (guided_next_start >= guided_end) {
            pthread_mutex_unlock(&guided_mutex);
            break;
        }

        // 引导调度：块大小 = 剩余任务数 / (2 * 线程数)
        int remaining = guided_end - guided_next_start;
        chunk = remaining / (2 * guided_num_threads);
        if (chunk < guided_increment) {
            chunk = guided_increment;
        }

        local_start = guided_next_start;
        local_end = local_start + chunk * guided_increment;
        if (local_end > guided_end) {
            local_end = guided_end;
        }
        guided_next_start = local_end;
        pthread_mutex_unlock(&guided_mutex);

        for (int i = local_start; i < local_end; i += guided_increment) {
            work->functor(i, work->arg);
        }
    }
    return NULL;
}
```

#### 2.3.5 单线程优化

```c
if (num_threads == 1) {
```

### 2.4 计时与验证

#### 2.4.1 墙上时间计时

```c
#define _POSIX_C_SOURCE 199309L
#include <time.h>

static inline double wtime(void) {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return ts.tv_sec + 1e-9 * ts.tv_nsec;
}

// 使用
double start = wtime();
```

**注意**：不能使用 `clock()`，因为多线程下 `clock()` 返回的是所有线程的 CPU 时间总和，会导致加速比完全失真。

#### 2.4.2 动态容差验证

```c
int verify_result(float *C_serial, float *C_parallel, int M, int K, int N) {
    float tolerance;
    if (N <= 512) {
        tolerance = 1e-3f;
    } else if (N <= 1024) {
        tolerance = 1e-2f;
    } else {
        tolerance = 1e-1f;
    }

    for (int i = 0; i < M * K; i++) {
        float diff = C_serial[i] - C_parallel[i];
        if (diff < 0) diff = -diff;
        if (diff > tolerance) {
            printf("验证失败: C[%d] = %f, 期望 %f, 差值 %f\n", i,
                   C_parallel[i], C_serial[i], diff);
```

2. 串行时间较短，线程创建和同步开销占比相对较高

## 4. 实验感想

### 4.1 遇到的问题及解决方案

#### 问题一：计时方式错误导致加速比失真

**现象**：最初使用 `clock()` 计时，多线程下加速比完全不合理（有时甚至小于 1）。

**分析**：
- `clock()` 返回的是所有线程的 CPU 时间总和
- 4 线程并行时，`clock()` 返回值 ≈ 4 × 实际墙上时间
- 算出来的"并行时间"比串行还长，加速比完全失真

**解决方案**：改用 `clock_gettime(CLOCK_MONOTONIC, ...)` 测量墙上时间（Wall-Clock Time）。

```c
// 错误方式
clock_t start = clock();
// ... 并行执行 ...
clock_t end = clock();
double time = (double)(end - start) / CLOCKS_PER_SEC;  // 多线程下错误！

// 正确方式
double start = wtime();
// ... 并行执行 ...
double end = wtime();
double time = end - start;  // 墙上时间，正确！
```

**结论**：并行程序计时必须使用墙上时间，不能使用 CPU 时间。

#### 问题二：B 矩阵列访问导致缓存不友好

**现象**：最初未使用 B 转置，加速比不理想。

**分析**：
- `B[p * K + j]` 在内层循环中是列优先访问
- 每次访问跳跃 K 个元素，缓存命中率低
- 对于大规模矩阵，B 矩阵可能超过 CPU 缓存容量

**解决方案**：预先转置 B 矩阵，将 `B[p][j]` 转换为 `Bt[j][p]`，实现连续内存访问。

### 4.2 对并行计算的理解

#### 4.2.1 OpenMP vs Pthreads

| 特性 | OpenMP | Pthreads |
|------|--------|----------|
| 编程难度 | 低（编译器指令） | 高（手动管理） |
| 性能 | 高（编译器优化） | 略低 |
| 灵活性 | 中 | 高 |
| 可移植性 | 高 | 中 |
| 适用场景 | 循环并行 | 复杂并行模式 |

**建议**：对于简单的循环并行，优先使用 OpenMP；对于复杂的并行模式（如动态任务分配），使用 Pthreads。

#### 4.2.2 调度策略的选择

| 调度策略 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| 静态 | 负载均匀 | 无锁，开销小 | 负载不均时效率低 |
| 动态 | 负载不均 | 负载均衡 | 锁竞争，开销大 |
| 引导 | 负载未知 | 兼顾两者 | 实现复杂 |

**本实验结论**：矩阵乘法每行计算量相同，负载均匀，静态调度最佳。

#### 4.2.3 缓存优化的重要性

B 矩阵转置优化展示了缓存优化对性能的巨大影响：

- **优化前**：列访问，缓存命中率约 30%
- **优化后**：行访问，缓存命中率约 90%+
- **效果**：单线程加速比从 ~1x 提升到 ~5.5x（1024×1024 矩阵）

**启示**：在并行化之前，应先优化串行代码的缓存友好性。

#### 4.2.4 超线性加速现象

1024×1024 矩阵 OpenMP 8 线程达到 8.47x 加速比，超过线性加速（8x）。

**可能原因**：
1. 多线程并行访问不同内存区域，减少了内存带宽瓶颈
2. 每个线程的局部性更好，缓存命中率更高
3. 现代 CPU 的乱序执行和预取机制在多线程下更有效

### 4.3 总结

本次实验通过实现 OpenMP 和 Pthreads 两种并行矩阵乘法，深入理解了并行计算的核心概念：

1. **矩阵乘法是计算密集型任务**：$O(N^3)$ 计算密度，非常适合并行化
2. **并行化效果取决于问题规模**：规模越大，加速比越高
3. **计时方式至关重要**：必须使用墙上时间，不能使用 CPU 时间
4. **缓存优化先于并行化**：B 矩阵转置带来巨大性能提升
5. **OpenMP 优于 Pthreads**：对于简单循环并行，OpenMP 性能更高、代码更简洁
6. **静态调度适合均匀负载**：矩阵乘法每行计算量相同，静态调度最佳
7. **单线程优化很重要**：线程数为1时直接串行执行，避免框架开销
